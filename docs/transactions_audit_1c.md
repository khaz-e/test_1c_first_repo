# Транзакции 1С (явные и неявные): правила для ИИ-агента

## 1. Назначение документа

Документ задаёт правила **написания и ревью** прикладного кода с **явными** транзакциями (`НачатьТранзакцию` / `ЗафиксироватьТранзакцию` / `ОтменитьТранзакцию`) и с учётом **неявных** транзакций платформы при записи в БД. Цель — целостность связанных данных и снижение риска трудноотлавливаемых ошибок вида **«В этой транзакции уже происходили ошибки»** / **«В данной транзакции происходила ошибка»**.

## 2. Правила для ИИ-агента: порядок проверки

1. **Нужна ли явная транзакция?** Запись объекта **сама** открывает транзакцию; не добавлять `НачатьТранзакцию` только ради одной записи без других действий с БД и без [ответственного чтения](https://its.1c.ru/db/v8std/content/648/hdoc) (§4).
2. **Один метод:** `НачатьТранзакцию` и `ЗафиксироватьТранзакцию` / `ОтменитьТранзакцию` — в **одном** методе, не разносить по стеку вызовов (§3).
3. **`Попытка`–`Исключение`:** `НачатьТранзакцию` **снаружи** `Попытка`; вся работа после начала — в **одном** `Попытка`; `ЗафиксироватьТранзакцию` **последним** в `Попытка` перед `Исключение`; в `Исключение` сначала **`ОтменитьТранзакцию`** (§5).
4. **Вложенный сценарий (п. 1.4 стандарта):** при вложенной логике в конце `Исключение` **рекомендуется** **`ВызватьИсключение`**, чтобы не сломать внешнюю транзакцию (§5).
5. **Активная транзакция с ошибкой:** не обращаться к БД из обработчика, если транзакция **ещё активна** (в т.ч. неявное обращение через **представление** ссылки) — §6.
6. **Длительность:** в транзакции только **неделимое** по бизнес-логике; ориентир порции в нагруженной ИС — **до ~20 с** на одну транзакцию (связь с таймаутом ожидания блокировки); не объединять в одну транзакцию раздельные по смыслу действия без веской причины (§7).
7. **Минимум внутри транзакции:** тяжёлые расчёты и запросы без блокировки — **до** транзакции; внешние сетевые вызовы — **не** в транзакции (§8).
8. **Итоги регистра сведений:** отключение итогов + запись + включение итогов — **обязательно** в одной явной транзакции; `ПересчитатьИтоги()` — как в примере стандарта (§9).

## 3. Назначение транзакции (смысл)

Транзакции обеспечивают **целостное** изменение **связанных** данных: все действия с БД в рамках транзакции либо выполняются **целиком**, либо **целиком откатываются**.

---

## 4. Особенности платформы (кратко из стандарта)

- **Вложенные транзакции не поддерживаются** (см. ИТС по ссылке выше).
- При **исключении** в общем случае транзакция **не может быть зафиксирована**, независимо от того, обработано исключение или нет.
- Явная транзакция — **`НачатьТранзакцию()`**; платформа **неявно** начинает транзакцию при **любой записи** в БД.

Несоблюдение правил ведёт к ошибкам **«В этой транзакции уже происходили ошибки»**, которые трудно воспроизвести.

---

## 5. Парность `НачатьТранзакцию` и завершения

Исключение **не отменяет** транзакцию немедленно, но **запрещает** успешное завершение. Вызовы **`НачатьТранзакцию`** с одной стороны и **`ЗафиксироватьТранзакцию`** / **`ОтменитьТранзакцию`** с другой должны быть **парными**.

---

## 6. Начало и фиксация в одном методе (п. 1.2)

Начало транзакции и её фиксация (отмена) должны выполняться в **одном** методе.

### Правильно (пример из стандарта)

```bsl
Процедура ЗаписатьДанныеВИБ()

	НачатьТранзакцию();

	Попытка
		// чтение или запись данных
		ДокументОбъект.Записать();
		ЗафиксироватьТранзакцию();
	Исключение
		ОтменитьТранзакцию();
		// дополнительные действия по обработке исключения
	КонецПопытки;

КонецПроцедуры
```

### Неправильно (пример из стандарта)

```bsl
Процедура ЗаписатьДанныеВИБ()
	НачатьТранзакцию();
	ЗаписатьДокумент();
КонецПроцедуры

Процедура ЗаписатьДокумент()

	Попытка
		// чтение или запись данных
		ДокументОбъект.Записать();
		ЗафиксироватьТранзакцию();
	Исключение
		ОтменитьТранзакцию();
		// дополнительные действия по обработке исключения
	КонецПопытки;

КонецПроцедуры
```

## 7. Структура `Попытка`–`Исключение` при явной транзакции (п. 1.3)

| Требование | Суть |
|------------|------|
| `НачатьТранзакцию` | **Вне** `Попытка`–`Исключение`, сразу **перед** `Попытка` |
| После начала транзакции | Все действия в **одном** блоке `Попытка` (чтение, блокировка, запись) |
| `ЗафиксироватьТранзакцию` | **Последним** в `Попытка` перед `Исключение`, чтобы после фиксации не возникло исключения |
| `Исключение` | Сначала **`ОтменитьТранзакцию`**, затем прочие действия |
| Журнал | **Рекомендуется** **`ЗаписьЖурналаРегистрации`** в `Исключение` ([стандарт 498](https://its.1c.ru/db/v8std/content/498/hdoc)) |
| Вложенная логика (п. 1.4) | В конце `Исключение` **рекомендуется** **`ВызватьИсключение`**, иначе внешняя транзакция не будет явно отменена и возможна ошибка **«В данной транзакции происходила ошибка»** |

### Пример из стандарта: блокировка, запись, журнал, проброс исключения

```bsl
НачатьТранзакцию();
Попытка
	БлокировкаДанных = Новый БлокировкаДанных;
	ЭлементБлокировкиДанных = БлокировкаДанных.Добавить("Документ.ПриходнаяНакладная");
	ЭлементБлокировкиДанных.УстановитьЗначение("Ссылка", СсылкаДляОбработки);
	ЭлементБлокировкиДанных.Режим = РежимБлокировкиДанных.Исключительный;
	БлокировкаДанных.Заблокировать();

	// чтение или запись данных
	ДокументОбъект.Записать();

	ЗафиксироватьТранзакцию();
Исключение
	ОтменитьТранзакцию();

	ЗаписьЖурналаРегистрации(
		НСтр("ru = 'Выполнение операции'"),
		УровеньЖурналаРегистрации.Ошибка,
		,
		,
		ОбработкаОшибок.ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));

	ВызватьИсключение; // есть внешняя транзакция
КонецПопытки;
```

## 8. Когда не открывать явную транзакцию (п. 1.4.1)

- Если кроме **записи объекта** других действий с БД **нет** — платформа при записи **сама** откроет транзакцию.
- Не открывать явную транзакцию, если не нужно [ответственное чтение данных](https://its.1c.ru/db/v8std/content/648/hdoc). Обычно оно **не** требуется при записи **нового** объекта (нового набора записей регистра).
- При **`ПолучитьОбъект`** / **`Прочитать`** (наборы) анализировать необходимость **ответственного** чтения и только тогда решать о **`НачатьТранзакцию`**.

### Правильно (пример из стандарта)

```bsl
Попытка
	ДокументОбъект = Документы.ПриходнаяНакладная.СоздатьДокумент();
	// действия по заполнению объекта
	ДокументОбъект.Записать();
Исключение
	// действия по обработке исключения
КонецПопытки;
```

### Неправильно (пример из стандарта)

```bsl
НачатьТранзакцию();
Попытка
	ДокументОбъект = Документы.ПриходнаяНакладная.СоздатьДокумент();
	// действия по заполнению объекта
	ДокументОбъект.Записать();
	ЗафиксироватьТранзакцию();
Исключение
	ОтменитьТранзакцию();
КонецПопытки;
```

### Метод только внутри уже открытой транзакции (п. 1.4.2)

Если метод вызывается **только** при уже открытой транзакции (например, из `ПередЗаписью`, `ОбработкаПроведения` и т.п.), явно открывать в нём транзакцию в общем случае **бессмысленно**.

### Вложенная логика ради текста ошибки (п. 1.4.3)

Стандарт допускает вложенную обработку для **улучшения формулировок** сообщений (пример в ИТС: электронная подпись vs запись файла).

### Нельзя трогать БД при ещё активной транзакции после ошибки (п. 1.4.4)

Если транзакция **ещё активна** (например, ошибка во **вложенной** транзакции), **нельзя** обращаться к БД — иначе **«В этой транзакции уже происходили ошибки»**. Учитывать **неявные** обращения (например, **представление** ссылки).

## 9. Длительность и состав транзакции (разд. 2 стандарта)

### 9.1 Неделимость

В одной транзакции — действия, **неделимые** по бизнес-логике.

**Пример из стандарта:** при проведении документа записываются документ и движения; если не прошла запись **хотя бы в один** регистр — всё проведение должно быть **отменено**.

Если действия **могут** выполняться по отдельности, их в общем случае **не следует** объединять в одну транзакцию.

**Исключение (оптимизация, п. 2.1.2):** несколько **несвязанных** объектов в одной транзакции — только с взвешенным выбором **порции** данных (баланс длительности, объёма фиксации и числа транзакций).

### 9.2 Долгие транзакции (п. 2.2)

**Неправильно (идея примера ИТС):** загрузка адресного классификатора — все данные одной версии (объём порядка **1 ГБ**) в **одной** транзакции ради отката версии целиком. Риски: нехватка **ОЗУ** (особенно файловая ИБ на **32-бит** ОС), долгое выполнение, нельзя ускорить **несколькими потоками**.

**Правильно (из стандарта):** разбить на порции так, чтобы запись порции в одной транзакции **не превышала ~20 секунд** в **высоконагруженной** ИС; откат к предыдущей версии при ошибке — **отдельной** функциональностью. **20 секунд** согласованы с временем ожидания установки транзакционной блокировки по умолчанию.

### 9.3 Ресурсы (п. 2.2.1)

В длинной транзакции: запись изменений в **журнал транзакций** СУБД; **блокировки** до конца транзакции; на сервере 1С блокировки в **ОЗУ**; прочие ресурсы по логике.

### 9.4 Параллельность (п. 2.2.2)

Пересечение по ресурсам → ожидание (**по умолчанию ~20 с**), затем **«Превышено время ожидания установки блокировки»**. Повод разобрать: вынести за транзакцию (п. **2.4** стандарта), оптимизировать алгоритм и блокировки.

### 9.5 Минимум работы внутри транзакции (п. 2.3)

- Тяжёлые расчёты — **до** транзакции, если позволяет логика.
- Расчёт в транзакции — максимально **простой**; пример стандарта: контроль остатков **после** записи **простым** запросом к регистру.
- Проверка заполнения — **вне** транзакции ([463](https://its.1c.ru/db/v8std/content/463/hdoc)).
- Запросы без необходимости блокировки — **до** транзакции ([648](https://its.1c.ru/db/v8std/content/648/hdoc)).
- Запросы в транзакции — оптимизировать (**`docs/query_language_audit_1c.md`**).
- **Не** вызывать в транзакции **внешние** ресурсы (сети, сайты, почта, FTP, HTTP, веб-сервисы и т.д.): доступность, тайм-аут, длительность.

## 10. Обязательная явная транзакция: итоги регистра сведений (п. 3.1)

Если для ускорения записи в регистр **отключаются итоги**, операцию **вместе** с отключением и включением итогов нужно выполнять **в транзакции** — иначе в других сеансах возможна ошибка при получении **среза последних**.

### Неправильно (пример из стандарта)

```bsl
РегистрыСведений.КурсыВалют.УстановитьИспользованиеИтогов(Ложь);
НаборЗаписей = РегистрыСведений.КурсыВалют.СоздатьНаборЗаписей();
НаборЗаписей.Отбор.Валюта.Установить(ВалютаСсылка);
НаборЗаписей.Загрузить(ТаблицаКурсов);
НаборЗаписей.ОбменДанными.Загрузка = Истина;
НаборЗаписей.Записать();
РегистрыСведений.КурсыВалют.УстановитьИспользованиеИтогов(Истина);
РегистрыСведений.КурсыВалют.ПересчитатьИтоги();
```

### Правильно (пример из стандарта)

```bsl
НачатьТранзакцию();
Попытка
	РегистрыСведений.КурсыВалют.УстановитьИспользованиеИтогов(Ложь);
	НаборЗаписей = РегистрыСведений.КурсыВалют.СоздатьНаборЗаписей();
	НаборЗаписей.Отбор.Валюта.Установить(ВалютаСсылка);
	НаборЗаписей.Загрузить(ТаблицаКурсов);
	НаборЗаписей.ОбменДанными.Загрузка = Истина;
	НаборЗаписей.Записать();
	РегистрыСведений.КурсыВалют.УстановитьИспользованиеИтогов(Истина);
	ЗафиксироватьТранзакцию();
Исключение
	ОтменитьТранзакцию();
	ВызватьИсключение;
КонецПопытки;
РегистрыСведений.КурсыВалют.ПересчитатьИтоги();
```

## 11. Чеклист самопроверки перед выдачей кода

1. Явная транзакция **не** дублирует то, что платформа уже делает при **одной** записи без ответственного чтения и без сопутствующих операций?
2. `НачатьТранзакцию` / `ЗафиксироватьТранзакцию` / `ОтменитьТранзакцию` **парны** и находятся в **одном** методе?
3. `НачатьТранзакцию` стоит **перед** `Попытка`, а `ЗафиксироватьТранзакцию` — **последним** в `Попытка`?
4. В `Исключение` сначала **`ОтменитьТранзакцию`**, при необходимости журнал и **`ВызватьИсключение`** для внешней транзакции?
5. Нет обращений к БД в обработчике при **активной** «битой» транзакции (в т.ч. через представление ссылки)?
6. Транзакция **короткая** по смыслу и по времени; нет сетевых/внешних вызовов внутри?
7. При отключении **итогов** регистра сведений для массовой записи — явная транзакция по **п. 3.1** и примеру выше?
8. При параллельной работе учтены **`docs/blocking_data_1c.md`** и длина транзакции — **`docs/excessive_locks_audit_1c.md`**?
