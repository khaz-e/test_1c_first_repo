# Правила разработки в расширении (dev rules)

## 1. Контекст решения

- Основное прикладное решение: **УТ 11.5.22.174**.
- Каталог `UT11KhazE` — эталон основной конфигурации для сверки логики и структуры.
- Доработки выполняются в расширении; в основную конфигурацию правки вносятся только при невозможности закрыть задачу в расширении и если это оговорено.

## 2. Обязательные правила

1. **Приоритет расширения.** Работать в рамках расширения; новые объекты создавать с проектным префиксом.
2. **Язык кода.** Использовать русский синтаксис 1С (если отдельно не оговорено иное).
3. **Стандарты.** Следовать стандартам 1С (ITS) и практикам БСП.
4. **Использование БСП.** Предпочитать типовые механизмы БСП вместо дублирования самописной логики.
5. **Контекст выполнения.** Явно выбирать директивы `&НаКлиенте`, `&НаСервере`, `&НаСервереБезКонтекста` и т.д.
6. **Запросы к БД (язык запросов 1С).** Тексты выборок пишутся на языке запросов платформы; типичные ловушки — соединение с подзапросом или виртуальной таблицей без ВТ, несоответствие условий индексам, **`ИЛИ` в `ГДЕ`**, подзапрос в **`ПО`**, обращение через точку от составного типа, отбор виртуальной таблицы только в **`ГДЕ`** вместо параметров таблицы. Чеклист и примеры — **`docs/query_language_audit_1c.md`** (ИТС 5842); избегать запросов в цикле.
7. **Формат файлов.** `.bsl` рассматривать как исходный код модулей 1С.
8. **Блокировки данных.** При серверной работе с БД в параллельных сеансах (проведение, регистры, «прочитал — затем изменил») соблюдать режим управления блокировкой и явные **`БлокировкаДанных`** / **`Заблокировать()`**; подробности, примеры и чеклист — в **`docs/blocking_data_1c.md`**. Тексты выборок из БД на языке запросов 1С — **`docs/query_language_audit_1c.md`**.
9. **Транзакции.** Явные **`НачатьТранзакцию`** / **`ЗафиксироватьТранзакцию`** / **`ОтменитьТранзакцию`** — только по необходимости (платформа сама открывает транзакцию при записи); пара с **`Попытка`**–**`Исключение`**, один метод, короткая транзакция, без лишних обращений к БД после ошибки; итоги регистра при массовой записи — по стандарту. Чеклист и примеры — **`docs/transactions_audit_1c.md`** (ИТС v8std 783).
10. **Избыточные блокировки.** Проверять код и метаданные на типичные причины лишних ожиданий (сначала неоптимальные запросы — **`docs/query_language_audit_1c.md`**, затем константы, последовательность, регистры, длинные транзакции, обмен) по **`docs/excessive_locks_audit_1c.md`**.
11. **Журнал регистрации.** Писать в журнал только при необходимости дополнительной диагностики для администратора; одна запись — одно событие; тип события в формате `Группа.Имя` без данных ИБ в строке типа; уровень и объём комментария по смыслу; не опираться на быстрые выборки из журнала. Подробности и примеры из ИТС v8std 498 — **`docs/event_log_usage_rules_1c.md`**.
12. **Часто используемые API БСП/УТ.** Описания, параметры и возвращаемые значения — по комментариям в эталоне `UT11KhazE` (`Module.bsl`); справочник и примеры из шпаргалки проекта — **`docs/frequently_used_bsp_api_reference_1c.md`**. Перечень для ориентира: **`ОбщегоНазначения`**, **`ОбщегоНазначенияКлиент`**, **`ОбщегоНазначенияКлиентСервер`**, **`Пользователи`**, **`ОрганизацииСервер`**, **`ПрефиксацияОбъектовКлиентСервер`**, **`ДополнительныеОтчетыИОбработки`**, **`ОбщегоНазначенияУТ`** / **`ОбщегоНазначенияУТКлиент`** / **`ОбщегоНазначенияУТКлиентСервер`**, **`СтроковыеФункцииКлиентСервер`** / **`СтроковыеФункции`** / **`СтроковыеФункцииКлиент`**, **`ФизическиеЛицаКлиентСервер`**, **`СклонениеПредставленийОбъектов`**, **`ПолучениеФайловИзИнтернета`**, **`ГенерацияШтрихкода`**, **`УправлениеПечатью`**, **`ФормированиеПечатныхФорм`**, **`НоменклатураКлиентСервер`**, **`КомпоновкаДанныхКлиентСервер`** / **`КомпоновкаДанныхСервер`**, **`ЗаказыСервер`**, **`РаботаСКонтрагентами`**, **`ПартнерыИКонтрагенты`**, **`ПроверкаКонтрагентов`**, **`УправлениеСвойствами`** / **`УправлениеСвойствамиСлужебный`**, **`ДлительныеОперации`**, **`СобытияФорм`**, **`Тарификация`**.

## 3. Структура кода модулей

Во всех программных модулях выделять код областями:

- `#Область ...`
- `#КонецОбласти`

Рекомендуемые области (по смыслу модуля):

- `ПрограммныйИнтерфейс`
- `СлужебныйПрограммныйИнтерфейс`
- `ОписаниеПеременных`
- `ОбработчикиСобытий`
- `СлужебныеПроцедурыИФункции`

### Комментарий

Единая структура областей упрощает поддержку, код-ревью и поиск типовых блоков в больших модулях.

## 4. Документация экспортных методов

Для **каждой экспортной** процедуры/функции перед объявлением обязателен комментарий в стиле БСП/ITS:

1. Назначение метода.
2. Блок `Параметры`:
   - имя параметра;
   - тип (или ожидаемые типы);
   - краткое назначение.
3. Для функций — блок `Возвращаемое значение`:
   - тип;
   - смысл результата, включая особые случаи (`Неопределено`, пустая коллекция и т.п.).
4. По возможности — блок `Пример`.
5. Для коллекций указывать **тип коллекции и тип элемента**.

### Мини-шаблон

```bsl
// Краткое назначение метода.
//
// Параметры:
//  Параметр1 - Строка - описание.
//  Параметр2 - Массив из Структура - описание.
//
// Возвращаемое значение:
//  Структура - описание результата.
//
// Пример:
//  Результат = ИмяФункции("строка", Новый Массив);
//
Функция ИмяФункции(Знач Параметр1, Знач Параметр2) Экспорт
	Возврат Новый Структура("Поле1, Поле2", Параметр1, Параметр2.Количество());
КонецФункции
```

## 5. Проверка структуры новых объектов

Если создается новый объект метаданных (форма, реквизит, документ, справочник и т.д.):

1. Проверить, есть ли в расширении аналогичный объект.
2. Сохранить единообразие:
   - типы данных;
   - подход к именованию;
   - используемые типовые модули/механизмы;
   - структура и стиль кода.
