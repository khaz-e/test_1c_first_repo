# Системные правила для ИИ-агента 1С (управляемые формы)

## 1. Назначение документа

Документ задает единый стандарт для разработки в управляемых формах 1С: где должен выполняться код (клиент/сервер), как передавать данные, как избегать потерь производительности и утечек памяти, и как оформлять типовые клиент-серверные сценарии.

## 2. Базовые принципы

1. **Клиент для UI, сервер для данных.**
   - Тонкий/веб-клиент не выполняет тяжелые вычисления и не работает с БД напрямую.
   - Весь код должен быть разделен директивами компиляции:
     - `&НаКлиенте`
     - `&НаСервере`
     - `&НаСервереБезКонтекста`
     - `&НаКлиентеНаСервереБезКонтекста`
2. **Явный контекст и детерминизм.**
   - Не полагаться на неявную передачу состояния формы.
   - Параметры в серверные вызовы передавать явно.
3. **`&НаСервереБезКонтекста` по умолчанию.**
   - `&НаСервере` использовать только когда действительно нужен весь контекст формы.
4. **Минимум данных на клиенте.**
   - На клиент передавать только данные, которые реально отображаются пользователю.
   - Не переносить на клиент «тяжелые» структуры без необходимости.
5. **Безопасность по умолчанию.**
   - Проверка прав, запись, транзакции, бизнес-правила — только на сервере.
   - Параллельный доступ к одним и тем же данным в БД (остатки, проведение, пакетная запись) — управляемые и автоматические блокировки, явный **`Заблокировать()`**; см. **`docs/blocking_data_1c.md`**. Тексты серверных выборок из БД на языке запросов 1С — **`docs/query_language_audit_1c.md`**.

## 3. Контекст формы и цена вызова

- Вызов `&НаСервере` (с контекстом) передает на сервер состояние формы, а затем возвращает его обратно.
- Вызов `&НаСервереБезКонтекста` передает только параметры и возвращает только результат.
- Каждый переход клиент ↔ сервер — сетевой round-trip; их нужно минимизировать.

### Практические правила оптимизации

1. Не вызывать сервер в цикле.
2. Для одного действия пользователя агрегировать операции в один серверный вызов.
3. Длительные **серверные** операции — через **длительные операции** и/или **фоновые задания**; клиентский **`Асинх`/`Ждать`** не отменяет необходимости выносить тяжёлую работу с клиента (см. **`docs/async_await_1c.md`**).
4. Пагинацию и отборы всегда выполнять на сервере (например, через параметры запроса/динамический список); при написании текста запроса — **`docs/query_language_audit_1c.md`**.

## 4. Данные формы и объекты БД

В клиентском коде формы объектные типы БД напрямую не используются как рабочая модель. Для формы используются данные формы (суррогаты), например:

- `ДанныеФормыСтруктура`
- `ДанныеФормыКоллекция`
- `ДанныеФормыЭлементДерева`

Если нужно выполнить серверные методы объекта (заполнение, запись и т.п.), используется преобразование:

```bsl
// Из данных формы получаем реальный объект для работы (имя реквизита — из метаданных формы).
ДокументОбъект = РеквизитФормыВЗначение("Объект");
// Выполняем серверные операции (аргументы — по прикладной логике и сигнатуре метода объекта).
ДокументОбъект.Заполнить(Неопределено);
// Возвращаем измененные данные обратно в форму
ЗначениеВРеквизитФормы(ДокументОбъект, "Объект");
```

> Примечание: `РеквизитФормыВЗначение`/`ЗначениеВРеквизитФормы` — ресурсоемкие операции, применять только когда без методов объекта не обойтись.

## 5. Передача данных и сериализация

### 5.1 Что обычно допустимо передавать

- Примитивы: `Число`, `Строка`, `Дата`, `Булево`, `Неопределено`, `Null`
- Идентификаторы и ссылки: `УникальныйИдентификатор`, ссылки на прикладные объекты
- Контейнеры: `Структура`, `Соответствие`, `Массив`
- В отдельных серверных сценариях: `ТаблицаЗначений`, `ДеревоЗначений` (см. ограничения ниже)

### 5.2 Что нельзя передавать напрямую

- `Запрос`, `РезультатЗапроса`
- объекты формы, динамические списки, менеджеры объектов
- `COMObject`, потоки, файловые дескрипторы и пр. несериализуемые объекты

### 5.3 Ограничения и рекомендации

1. Даже когда тип формально сериализуем, **не передавать большой объем напрямую**: использовать временное хранилище.
2. Не возвращать `ТаблицаЗначений` с неописанными/неоднозначными типами колонок.
3. Для сложных результатов использовать `Структура`/`Соответствие` с явными полями.

## 6. Временное хранилище (ВХ)

### 6.1 Когда использовать

- передача тяжелых данных (файлы, картинки, большие таблицы)
- передача объектов, которые неудобно/небезопасно гонять напрямую между клиентом и сервером

### 6.2 Критичное правило памяти

При `ПоместитьВоВременноеХранилище` и родственных операциях всегда передавать идентификатор владельца (обычно `ЭтаФорма.УникальныйИдентификатор` или заранее созданный адрес), чтобы данные очищались автоматически и не копили RAM на сервере.

## 7. Асинхронность в управляемых формах

1. **Где смотреть полный стандарт.** Синтаксис и поведение **`Обещание`**, **`Асинх`**, **`Ждать`**, типовые ошибки (в т.ч. с параметром **`Отказ`**), версии платформы и примеры — в **`docs/async_await_1c.md`**. В этом разделе — только краткое напоминание для форм.
2. **Не распространять `Асинх`/`Ждать` на весь код.** Использовать только там, где это **реально** решает задачу лучше альтернатив: запрет синхронных вызовов, длинная цепочка клиентских ожиданий без дробления на колбэки, необходимость линейного чтения сценария и т.п. Если достаточно обычного клиент-серверного вызова или типового паттерна БСП — **не усложнять**.
3. **Как ИИ-агенту предлагать асинхронность.** Упоминание **`Асинх`/`Ждать`** допускается **только** в конце ответа / в плане решения отдельным пунктом и **только** с явным перечислением **преимуществ для конкретной задачи**. Нет устойчивого преимущества — **не предлагать**.
4. **Длительные серверные операции** (отчёты, массовая запись, тяжёлый HTTP) — по правилам конфигурации: **длительные операции**, фоновые задания, а не «асинхронный сервер ради ключевого слова»; пересечение с клиентским `Асинх`/`Ждать` см. **`docs/async_await_1c.md`**.

**Пример: вопрос перед закрытием на `ВопросАсинх` + `Ждать`** (обработчик формы — обычная процедура, чтобы корректно работал **`Отказ`**). В модуле формы объявите **`Перем ВопросПриЗакрытииЗадан;`** и при инициализации присвойте **`Ложь`**.

```bsl
&НаКлиенте
Перем ВопросПриЗакрытииЗадан;

// Инициализация — в клиентской процедуре ПриОткрытии: ВопросПриЗакрытииЗадан = Ложь;

&НаКлиенте
Процедура ПередЗакрытием(Отказ, ЗавершениеРаботы, ТекстПредупреждения, СтандартнаяОбработка)

	Если НЕ ВопросПриЗакрытииЗадан Тогда
		ЗадатьВопросПриЗакрытииАсинх();
		Отказ = Истина;
	КонецЕсли;

КонецПроцедуры

&НаКлиенте
Асинх Процедура ЗадатьВопросПриЗакрытииАсинх()

	Результат = Ждать ВопросАсинх(
		НСтр("ru = 'Закрыть форму?'"),
		РежимДиалогаВопрос.ДаНет,
		,
		КодВозвратаДиалога.Да);

	Если Результат = КодВозвратаДиалога.Да Тогда
		ЭтаФорма.Закрыть();
	КонецЕсли;

	ВопросПриЗакрытииЗадан = Истина;

КонецПроцедуры
```

Переменная модуля **`ВопросПриЗакрытииЗадан`**, инициализация и полный разбор паттерна — в **`docs/async_await_1c.md`**, п. 11.

**Устаревший, но ещё встречающийся стиль** — `ПоказатьВопрос` + `ОписаниеОповещения`; при запрете модальности и на поддерживаемых версиях предпочтительнее **`ВопросАсинх`** (см. тот же файл).

## 8. Шаблоны (из исходных правил)

### 8.1 Базовый клиент-серверный вызов

```bsl
&НаКлиенте
Процедура ДействиеКлиент(Команда)
	Параметры = Новый Структура("Пар1, Пар2", 1, "тест");
	Результат = ДействиеНаСервере(Параметры);
	Элементы.Реквизит1.Значение = Результат.НовоеЗначение;
КонецПроцедуры

&НаСервереБезКонтекста
Функция ДействиеНаСервере(Знач Параметры)
	Возврат Новый Структура("Успех, Ошибка, НовоеЗначение", Истина, "", Параметры.Пар1 + СтрДлина(Параметры.Пар2));
КонецФункции
```

### 8.2 Загрузка файла на сервер через ВХ

Клиентский выбор файла и помещение во временное хранилище с **`Ждать`** — только внутри **`Асинх`**-процедуры. Ниже — **валидный** ориентир для платформы **8.3.18+** (имена методов и конструкторов сверять со справкой). Подробности, альтернативы (`ВыбратьАсинх` у диалога и т.д.) — в **`docs/async_await_1c.md`**.

```bsl
&НаКлиенте
Асинх Процедура ЗагрузитьФайлИВызватьСервер()

	ПараметрыДиалога = Новый ПараметрыДиалогаПомещенияФайлов(
		НСтр("ru = 'Выберите картинку'"),
		Ложь,
		"Картинки (*.jpg)|*.jpg",
		0);

	Обещание = ПоместитьФайлНаСерверАсинх(,,, ПараметрыДиалога, ЭтаФорма.УникальныйИдентификатор);
	Результат = Ждать Обещание;

	Если Результат = Неопределено Тогда
		Возврат;
	КонецЕсли;

	ОбработатьФайлНаСервере(Результат.Адрес);

КонецПроцедуры

&НаСервереБезКонтекста
Процедура ОбработатьФайлНаСервере(Знач АдресХранилища)
	ДвоичныеДанныеФайла = ПолучитьИзВременногоХранилища(АдресХранилища);
	// Дальнейшая обработка
КонецПроцедуры
```

### 8.3 Передача ТЗ во внеконтекстный серверный вызов через ВХ

Подготовка данных выполняется в контексте формы (**`&НаСервере`**), адрес ВХ возвращается на клиент как **результат функции** (присваивание «через» параметр вызова на клиенте для серверной процедуры без возврата значения в 1С не заполняет переменную клиента).

```bsl
&НаСервере
Функция ПодготовитьДанныеНаСервере()

	ТЗ = Новый ТаблицаЗначений;
	ТЗ.Колонки.Добавить("Товар");
	ТЗ.Колонки.Добавить("Количество");

	НоваяСтрока = ТЗ.Добавить();
	НоваяСтрока.Товар = Справочники.Номенклатура.НайтиПоНаименованию("Дрель");
	НоваяСтрока.Количество = 10;

	Возврат ПоместитьВоВременноеХранилище(ТЗ, ЭтаФорма.УникальныйИдентификатор);

КонецФункции

&НаКлиенте
Процедура ВыполнитьРасчет(Команда)

	АдресДанных = ПодготовитьДанныеНаСервере();
	РезультатРасчета = БыстрыйРасчетНаСервере(АдресДанных);

КонецПроцедуры

&НаСервереБезКонтекста
Функция БыстрыйРасчетНаСервере(Знач АдресДанных)

	ТаблицаДляРасчета = ПолучитьИзВременногоХранилища(АдресДанных);
	Возврат "Расчет окончен, обработано строк: " + ТаблицаДляРасчета.Количество();

КонецФункции
```

### 8.4 Пакетная обработка без вызовов сервера в цикле

```bsl
&НаКлиенте
Процедура ОбработатьПакет(Команда)

	Пакет = Новый Массив;
	// ТаблицаФормы — реквизит/табличная часть в данных формы; имя и источник строк подставьте по метаданным.
	Для Каждого СтрокаТаблицы Из ТаблицаФормы Цикл
		Пакет.Добавить(СтрокаТаблицы.Ссылка);
	КонецЦикла;

	ОбработатьПакетНаСервере(Пакет);

КонецПроцедуры

&НаСервереБезКонтекста
Процедура ОбработатьПакетНаСервере(Знач ПакетСсылок)
	Для Каждого Ссылка Из ПакетСсылок Цикл
		// Одна транзакция на пакет или батчевая запись; при конкуренции за одни данные — см. docs/blocking_data_1c.md
	КонецЦикла;
КонецПроцедуры
```

## 9. Запреты (антипаттерны)

1. `&НаСервере` по умолчанию без необходимости.
2. Серверные вызовы в цикле.
3. `Выполнить()` и динамический код в бизнес-логике.
4. Передача `Запрос`/`РезультатЗапроса` между контекстами.
5. Клиентская реализация финальных бизнес-проверок.
6. Отсутствие очистки данных ВХ.
7. Псевдосинтаксис асинхронности (`Ожидать`, `Асинхронная Процедура`, вымышленные директивы вроде `&НаСервереБезКонтекстаАсинх`) вместо документированных **`Асинх`**/**`Ждать`** — см. **`docs/async_await_1c.md`**.

## 10. Чек-лист самопроверки перед выдачей кода

1. Директива у каждой процедуры выбрана корректно?
2. Можно ли заменить `&НаСервере` на `&НаСервереБезКонтекста`?
3. Параметры и результат сериализуемы?
4. Нет вызовов сервера в цикле?
5. Для клиентских ожиданий UI при запрете синхронных вызовов рассмотрен асинхронный вариант (`Асинх`/`Ждать` или типовой механизм), если это уместно задаче (см. **`docs/async_await_1c.md`**, п. 2)?
6. Права/запись/транзакции — только сервер? При конкурентном доступе к данным БД учтены блокировки (**`docs/blocking_data_1c.md`**)?
7. Исключения и пользовательские сообщения обработаны корректно?
8. Для ВХ задан идентификатор/адрес для контроля времени жизни данных?
9. Для новых возможностей платформы указана минимальная версия, если это важно?
