# Блокировки данных 1С (БД, транзакции): правила для ИИ-агента

## 1. Назначение документа

Документ задаёт правила для генерации и ревью **серверного** кода, который читает и изменяет данные в информационной базе при **параллельной** работе нескольких сеансов. Используйте его при проведении документов, записи наборов регистров, пакетных операциях и любых сценариях «сначала прочитал — потом изменил одни и те же данные».

**Связанные материалы:**

- Официальная база (модель платформы, таблицы режимов, учебные листинги): [ИТС, материал 5839](https://its.1c.ru/db/metod8dev/content/5839/hdoc) (фрагмент книги по 8.1; имена методов и параметры для **вашей** версии платформы сверяйте с **синтакс-помощником**).
- **Язык запросов 1С** (тексты выборок из БД, индексы, ВТ, виртуальные таблицы): **`docs/query_language_audit_1c.md`** (ИТС [5842](https://its.1c.ru/db/metod8dev/content/5842/hdoc)).
- **Транзакции** (явные `НачатьТранзакцию`, `Попытка`–`Исключение`, длительность, итоги регистра сведений): **`docs/transactions_audit_1c.md`** (ИТС [v8std 783](https://its.1c.ru/db/v8std/content/783/hdoc)).
- **Избыточные блокировки** (что проверять до углубления в СУБД: запросы, константы, последовательность, регистры, транзакции, обмен): **`docs/excessive_locks_audit_1c.md`** (по ИТС [5841](https://its.1c.ru/db/metod8dev/content/5841/hdoc)).

**Не входит в объём:** разбор блокировок на уровне конкретной СУБД в мониторинге, настройка СУБД, полный курс по изоляции транзакций.

## 2. Правила для ИИ-агента: с чего начать

1. **Не путать два механизма.** **Объектные** блокировки (`Заблокировать()` у объекта справочника/документа в памяти) и **транзакционные** (СУБД + при управляемом режиме — менеджер 1С) решают **разные** задачи; термин «блокировка» в каждом случае означает своё.
2. **Узнать режим конфигурации.** Свойство **«Режим управления блокировкой данных»** (автоматический / управляемый / автоматический и управляемый) и режим на **объектах метаданных** определяют, будет ли участвовать **`БлокировкаДанных`** в транзакции и не возникнет ли исключение при вложенных транзакциях (см. §6).
3. **В управляемом режиме не полагаться на `ДЛЯ ИЗМЕНЕНИЯ` в запросе** как на замену управляемой блокировке — в управляемом режиме конструкция **не даёт того же эффекта**, что в автоматическом; нужна **исключительная** управляемая блокировка **до** запроса (см. §5 и §7). Текст самого запроса к БД при этом должен быть оптимален по языку запросов — **`docs/query_language_audit_1c.md`**.
4. **Точка согласования сеансов — `Заблокировать()`.** По рабочей памятке (`Подвох…`): без явного вызова **`БлокировкаДанных.Заблокировать()`** внутри транзакции перед критичным чтением нельзя рассчитывать на то, что другой сеанс «подождёт» в смысле управляемых правил; «один только» текст запроса недостаточен. По ИТС запрос к данным проходит **менеджер 1С** при активном управляемом режиме; для прикладной дисциплины всё равно ориентируйтесь на явные блокировки там, где в конспекте ИТС показаны листинги.
5. **Сужать область блокировки.** Условия элементов **`БлокировкаДанных`** должны совпадать с условиями запроса (измерения, период, отборы); не подставлять целиком тяжёлую ТЧ, если в запросе узкий отбор (пример — листинг с `Услуга = ЛОЖЬ` в §7).
6. **Избыточные блокировки.** Если проблема — ожидания и «лишние» блокировки без явной бизнес-необходимости, проверять **до** детального разбора СУБД: неоптимальные запросы (ИТС 5842, правила агента **`docs/query_language_audit_1c.md`**), длинные транзакции, константы, последовательность, регистры, обмен, автоматический режим; чеклист и примеры — в **`docs/excessive_locks_audit_1c.md`**.
7. **Язык запросов 1С.** Любой текст выборки из БД (остатки, движения, справочники) проверять на типичные причины плохого плана и лишних блокировок СУБД: соединения с подзапросом/виртуальной таблицей, индексы, **`ИЛИ`**, подзапрос в **`ПО`**, точка от составного типа, параметры виртуальных таблиц — **`docs/query_language_audit_1c.md`**.

## 3. Объектные блокировки (кратко)

- **Пессимистическая:** `Объект.Заблокировать()` / `Разблокировать()` на **том же экземпляре**; в формах часто ставится при начале редактирования.
- **Важно:** одна только блокировка в памяти **не запрещает** другому сеансу менять запись в БД, если тот **не** вызывает блокировку того же объекта. Для согласования через объектные блокировки **оба** сеанса должны участвовать в той же дисциплине; конфликт — через **`Попытка … Исключение`** (исключение **не обязано** откатывать транзакцию).
- **Оптимистическая:** проверка **версии** перед записью; при расхождении — отказ в записи.

Транзакционные правила ниже применяются к **регистрам, запросам остатков, проведению** и т.п., а не к замене объектного `Заблокировать()` в форме.

---

## 4. Транзакции и зачем блокировать

**Транзакция** — последовательность операций с БД «всё или ничего». Параллельные транзакции без дисциплины чтения/записи дают классические проблемы: потерянное обновление, «грязное» чтение, неповторяемое чтение, фантомы (смысл — в любом учебнике по БД).


**Автоматический** режим транзакционных блокировок (как в 8.0): платформа опирается на **СУБД**, уровни изоляции **высокие** (по ИТС, ориентиры):

| СУБД | Вид блокировок (по ИТС) | Уровень изоляции |
|------|-------------------------|------------------|
| Файловая ИБ | Таблиц | Serializable |
| MS SQL Server | Записей | Repeatable Read или Serializable |
| IBM DB2 | Записей | Serializable |
| PostgreSQL | Таблиц | Read Committed |

Минус: **избыточные** блокировки, хуже параллелизм; возможна блокировка **крупного фрагмента** (в т.ч. по плану запроса — до уровня таблицы), из‑за чего страдают **другие** строки того же хранилища (см. **`docs/ПодвохСУправляемымиДанными.txt`**).

## 5. Управляемые транзакционные блокировки

### 5.1 Практическая модель («подвох»)

- Параллельный доступ к **одним и тем же** данным (например, остаток одной номенклатуры) требует явного согласования.
- Управляемые блокировки учитываются **менеджером 1С**; область задаётся **точечно** (пространства блокировок по метаданным). На СУБД при этом остаются **более мягкие** блокировки; в формулировке памятки «обычный» запрос без участия в той же дисциплине **не заменяет** явную установку блокировки для ожидания другого сеанса.
- **Аналогия с мьютексом:** критичный момент — вызов **`БлокировкаДанных.Заблокировать()`** **внутри транзакции** **до** запроса, который должен видеть согласованный снимок и не дать конкуренту испортить данные до конца вашей транзакции. В момент **`Заблокировать()`** при конфликте сеанс **ждёт** (до таймаута ожидания блокировки).
- **Разделяемый** режим: несколько транзакций могут **читать** под согласованными правилами; чужая транзакция с управляемыми блокировками **не должна изменять** заблокированные поля (согласованное чтение).
- **Исключительный** режим: для сценария изменения; **запрет чтения** под совместимыми управляемыми блокировками у других — ближе к смыслу **`ДЛЯ ИЗМЕНЕНИЯ`** в автоматическом режиме; также уменьшает риск **взаимной блокировки (deadlock)** в сценариях «все прочитали разделяемо — все пытаются писать» (по смыслу учебного примера ИТС).

### 5.2 Формулировки ИТС (дополнение)

- Менеджер транзакционных блокировок **1С** работает **наряду** с СУБД; при записи из встроенного языка платформа может ставить **свои** управляемые блокировки; разработчик добавляет **явные**, где нужно.
- **Порядок по ИТС:** обращение к данным проходит через **менеджер 1С** (автоматические и явные управляемые блокировки), затем выполнение в СУБД с **более низкой** изоляцией, чем в чисто автоматическом режиме для того же сценария.

## 6. Режим конфигурации и вложенные транзакции

**Значения свойства конфигурации «Режим управления блокировкой данных»:**

- **Автоматический** — в транзакции для выбора режима по этому глобальному свойству менеджер управляемых блокировок **не** используется (как 8.0).
- **Управляемый** — вся конфигурация в управляемом режиме в транзакции.
- **Автоматический и управляемый** — режим задаётся **на объект** метаданных (точечный перевод документов/регистров).

**Вложенные транзакции (таблица из ИТС):**

| Уже открыта | Начинается | Результат |
|-------------|------------|-----------|
| Автоматический | Автоматический | новая — автоматический |
| Управляемый | Управляемый | новая — управляемый |
| Автоматический | Управляемый | новая выполнится как **автоматический** |
| Управляемый | Автоматический | **исключение** |

**Примеры согласованности метаданных (ИТС):** если явная транзакция открыта в **управляемом** режиме, у записываемого объекта в метаданных тоже должен быть **управляемый** режим — иначе исключение. При проведении документа: если у документа автоматический режим — наборы регистров пишутся автоматически даже при ином свойстве регистра; если у документа управляемый — у регистров движений тоже должен быть **управляемый**.

**Параметр транзакции:** в материале ИТС для **`НачатьТранзакцию()`** упоминается параметр режима блокировок (в т.ч. значение по умолчанию **Автоматический**). В **вашей** версии платформы имя и наличие параметров проверьте в **синтакс-помощнике** (API эволюционировал после 8.1).

## 7. Объект `БлокировкаДанных`: API

```bsl
// Общая схема: коллекция элементов блокировки, затем фиксация одним вызовом.
БлокировкаДанных = Новый БлокировкаДанных;
ЭлементБлокировки = БлокировкаДанных.Добавить("ИмяПространстваБлокировок"); // например РегистрНакопления.ОстаткиТоваров.НаСкладах
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Разделяемый; // или Исключительный
ЭлементБлокировки.УстановитьЗначение("Измерение1", Значение1);
// Опционально: диапазон для даты/числа (границы включены)
// ЭлементБлокировки.УстановитьЗначение("Период", Новый Диапазон(ДатаНачала, ДатаКонца));
// Опционально: источник для множества значений измерения
// ЭлементБлокировки.ИсточникДанных = ТаблицаЗначенийИлиРезультатЗапроса;
// ЭлементБлокировки.ИспользоватьИзИсточникаДанных("Номенклатура", "Номенклатура");

БлокировкаДанных.Заблокировать(); // обязательно ВНУТРИ транзакции; иначе блокировки не установятся
```

**Пространства блокировок (идея):**

- Для ссылочных объектов: `Справочник.Имя`, `Документ.Имя`, … поле чаще всего **`Ссылка`**.
- Для регистров: вариант **`.НаборЗаписей`** (блокировка набора, поле **`Регистратор`**) и вариант **без суффикса** (остатки, «анализ», границы последовательности — по синтакс-помощнику и метаданным).

Управляемая блокировка в терминах **метаданных** **не равна** «строке таблицы СУБД» один к одному.

## 8. Примеры кода (учебные имена из ИТС; в проекте заменить на свои)

Имена **`ОстаткиНоменклатуры`**, **`Продажи`**, **`РасходнаяНакладная`** — из **статьи ИТС**, не из УТ.

### 8.1 Исключительная блокировка по одному измерению

```bsl
// Зачем: перед чтением остатков/оборотов в управляемом режиме зафиксировать исключительный доступ
// к срезу данных по складу, чтобы конкурент с управляемыми блокировками не прочитал и не изменил
// пересекающиеся данные в том же смысле, что описан в ИТС для листинга 1.

БлокировкаДанных = Новый БлокировкаДанных;
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.ОстаткиНоменклатуры");
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
ЭлементБлокировки.УстановитьЗначение("Склад", Склад);

БлокировкаДанных.Заблокировать();

// Далее — запрос остатков и запись движений в ТОЙ ЖЕ транзакции.
```

### 8.2 Разделяемая блокировка и диапазон периода

```bsl
// Зачем: согласованное чтение продаж по контрагенту за интервал дат без запрета параллельного
// чтения тем же режимом; запрет относится к ИЗМЕНЕНИЮ затронутых данных другими управляемыми транзакциями.

БлокировкаДанных = Новый БлокировкаДанных;
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.Продажи");
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Разделяемый;
ЭлементБлокировки.УстановитьЗначение("Контрагент", Контрагент);
Диапазон = Новый Диапазон(НачалоМесяца(Дата), Дата);
ЭлементБлокировки.УстановитьЗначение("Период", Диапазон);

БлокировкаДанных.Заблокировать();
```

### 8.3 Источник данных — табличная часть (узкий состав строк)

```bsl
// Зачем: заблокировать набор номенклатур из ТЧ одним элементом, не перечисляя в цикле.
// Источник — табличная часть или таблица значений с колонкой, совпадающей с именем поля пространства блокировки.

БлокировкаДанных = Новый БлокировкаДанных;
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.ОстаткиНоменклатуры");
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
ЭлементБлокировки.УстановитьЗначение("Склад", Склад);
ЭлементБлокировки.ИсточникДанных = СписокНоменклатуры; // коллекция строк с полем Номенклатура
ЭлементБлокировки.ИспользоватьИзИсточникаДанных("Номенклатура", "Номенклатура");

БлокировкаДанных.Заблокировать();
```

### 8.4 Источник из запроса (совпадение отбора с будущим запросом остатков)

```bsl
// Зачем: не блокировать «всю ТЧ», если в запросе остатков участвуют только строки с Номенклатура.Услуга = ЛОЖЬ.
// Условия блокировки должны совпадать с условиями запроса (требование ИТС).

БлокировкаДанных = Новый БлокировкаДанных;
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.ОстаткиНоменклатуры");
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;
ЭлементБлокировки.УстановитьЗначение("Склад", Склад);

ЗапросИсточник = Новый Запрос;
ЗапросИсточник.Текст =
"ВЫБРАТЬ РАЗЛИЧНЫЕ
|	РасходнаяНакладнаяСписокНоменклатуры.Номенклатура КАК Номенклатура
|ИЗ
|	Документ.РасходнаяНакладная.СписокНоменклатуры КАК РасходнаяНакладнаяСписокНоменклатуры
|ГДЕ
|	РасходнаяНакладнаяСписокНоменклатуры.Ссылка = &Ссылка
|	И РасходнаяНакладнаяСписокНоменклатуры.Номенклатура.Услуга = ЛОЖЬ";
ЗапросИсточник.УстановитьПараметр("Ссылка", Ссылка);

ИсточникНоменклатуры = ЗапросИсточник.Выполнить(); // в учебнике ИТС так; при необходимости Выгрузить() — по справке для вашей версии

ЭлементБлокировки.ИсточникДанных = ИсточникНоменклатуры;
ЭлементБлокировки.ИспользоватьИзИсточникаДанных("Номенклатура", "Номенклатура");

БлокировкаДанных.Заблокировать();

// Далее по ИТС — текст запроса остатков с соединением к виртуальной таблице Остатки(...) и запись движений;
// полный текст в статье на ИТС (в HTML фрагмент был сокращён многоточием).
```

## 9. Чеклист самопроверки перед выдачей кода

1. Определён ли **режим** блокировок конфигурации и объектов, с которыми работает код?
2. Для сценария «прочитал регистр / остатки — затем пишу» есть ли **`Заблокировать()` до чтения** с теми же измерениями/отборами, что и у запроса?
3. Выбран ли режим **Разделяемый** vs **Исключительный** осознанно (чтение vs запрет чтения, риск deadlock)?
4. **`Заблокировать()`** вызывается **внутри** транзакции, а не «для порядка» снаружи?
5. В управляемом режиме не рассчитываете ли вы на **`ДЛЯ ИЗМЕНЕНИЯ`** вместо **исключительной** блокировки?
6. Не блокируете ли вы **лишние** данные (широкий источник вместо узкого запроса)?
7. При вложенных транзакциях нет ли сочетания **внешняя управляемая + внутренняя автоматическая** (исключение по ИТС)?

---

## 10. Антипаттерны

1. Длинные транзакции с широкими **исключительными** блокировками «на всякий случай».
2. Блокировка без соответствия условиям **фактического** запроса (лишние измерения, другой склад, другой период).
3. Ожидание, что клиентская форма «сама» создаст транзакционные блокировки регистров при работе с динамическим списком без серверной процедуры проведения/записи.
4. Подмена изучения **синтакс-помощника** устаревшими примерами из статьи 8.1 без проверки версии.
