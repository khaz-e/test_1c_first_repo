# Язык запросов 1С (выборка из БД): правила для ИИ-агента

## 1. Назначение документа

Документ задаёт правила **написания и ревью** текстов запросов на **языке запросов 1С** к данным информационной базы: соединения, виртуальные таблицы, индексы, условия, составные типы. Цель — снизить риск неоптимального выполнения и **избыточных блокировок** (неоптимальные запросы в ИТС 5841 названы **главной** частой таких блокировок).

**Источник методики:** [ИТС, материал 5842](https://its.1c.ru/db/metod8dev/content/5842/hdoc). Практические оговорки по замерам, ВТ, периоду остатков и критериям отбора (в т.ч. из [доклада / статьи Infostart 548778](https://infostart.ru/1c/articles/548778/), 2016) встроены в **§3–§5, §8–§10** настоящего документа.

**Связанные документы:**

- Статистики СУБД: [ИТС 5837](https://its.1c.ru/db/metod8dev/content/5837/hdoc) (полное обновление статистик — влияет на стабильность планов после переписывания запросов).
- **Избыточные блокировки** (после оптимизации запроса всё ещё ожидания — константы, последовательность, регистры, транзакции, обмен): **`docs/excessive_locks_audit_1c.md`** (ИТС [5841](https://its.1c.ru/db/metod8dev/content/5841/hdoc)).
- Блокировки данных и **`БлокировкаДанных`**: **`docs/blocking_data_1c.md`**, [ИТС 5839](https://its.1c.ru/db/metod8dev/content/5839/hdoc).

## 2. Правила для ИИ-агента: порядок проверки запроса

1. **Соединения:** нет ли **`СОЕДИНЕНИЕ` с подзапросом** или с **виртуальной таблицей** при плохой производительности — переписать через **ВТ** и **пакетный запрос** (§3–§4).
2. **Индексы:** для условий в **`ГДЕ`**, **`ПО`**, параметрах виртуальной таблицы, **`ИМЕЮЩИЕ`** есть ли **подходящий** индекс (§5).
3. **`ИЛИ`:** не использовать в **`ГДЕ`** без необходимости; учесть **RLS** и несколько ролей; осторожно с **`ИЛИ` в `ПО`** (§6).
4. **Подзапрос в `ПО`:** запрещён по методике — вынос во ВТ (§7).
5. **Составной тип и точка:** не тянуть **`Поле.Реквизит`** без необходимости; **`ВЫРАЗИТЬ`**, дублирование в реквизит, сужение типов в метаданных (§8).
6. **Виртуальные таблицы:** отборы — в **параметры** таблицы, а не «фильтр только в `ГДЕ`» вне параметров (§9); для остатков на **актуальную** дату учитывать влияние параметра **периода** на план (итоги vs итоги+движения) — §9 и §10.
7. **Блокировки:** после проверки запроса при ожиданиях сеансов пройти **`docs/excessive_locks_audit_1c.md`**.
8. **ВТ и пакеты — не автоматическое ускорение:** накладные на **`ПОМЕСТИТЬ`** и **`ИНДЕКСИРОВАТЬ`** могут перевесить выигрыш на **малых** объёмах или в «простом» сценарии; итог зависит от **данных** — сравнивать замером (§3–§4, §10).
9. **Критерий отбора:** как способ получить на СУБД индекс под выборку из ТЧ — по ИТС; прямое использование в **тексте запроса** для выборки из документов **разных видов** часто даёт более тяжёлый SQL, чем явное **`ОБЪЕДИНЕНИЕ`** — проверять план (§5, §10).
10. **Замеры и среда:** оценивать стоимость исполнения, **I/O** и **CPU** на СУБД (для MS SQL — план и метрики в **«Консоли запросов для управляемого приложения 8.3»** с диска ИТС); оптимизировать в тех же условиях, что и рабочая система (§10).

## 3. Соединения с подзапросами

### Правило (ИТС)

- **Не** соединять с **подзапросом** — соединять только **объекты метаданных** или **временные таблицы**.
- Подзапрос в **любой** стороне соединения (`ЛЕВОЕ`, `ПРАВОЕ` и т.д.) — **потенциально опасно**.

### Последствия (из статьи)

- Очень медленное выполнение (до **порядков** разницы), нестабильность по условиям, разброс между **СУБД**, чувствительность к **статистикам** ([ИТС 5837](https://its.1c.ru/db/metod8dev/content/5837/hdoc)).

### Пример опасного фрагмента (как в ИТС)

```1c
ВЫБРАТЬ ...
ИЗ Документ.РеализацияТоваровУслуг
ЛЕВОЕ СОЕДИНЕНИЕ (
   ВЫБРАТЬ ИЗ РегистрСведений.Лимиты
   ГДЕ ...
   СГРУППИРОВАТЬ ПО ...
) ПО ...
```

### Методика

- Разбить на запросы по числу подзапросов в соединениях, объединить в **один пакет**; **индексировать ВТ** по **всем** полям условия **`ПО`**.
- Временные таблицы — с версии **8.1**; на **8.0** — обновление платформы.

### Пример пакета с `МенеджерВременныхТаблиц` (из статьи)

```bsl
// Создать менеджер временных таблиц
МенеджерВТ = Новый МенеджерВременныхТаблиц;
Запрос = Новый Запрос;
Запрос.МенеджерВременныхТаблиц = МенеджерВТ;
// Текст пакетного запроса
Запрос.Текст = "
  // Заполняем временную таблицу. Запрос к регистру лимитов.
   | ВЫБРАТЬ ...
   | ПОМЕСТИТЬ Лимиты
   | ИЗ РегистрСведений.Лимиты
   | ГДЕ ...
   | СГРУППИРОВАТЬ ПО ...
   | ИНДЕКСИРОВАТЬ ПО ...;
  
  // Выполняем основной запрос с использованием временной таблицы
   ВЫБРАТЬ ...
   ИЗ Документ.РеализацияТоваровУслуг
   ЛЕВОЕ СОЕДИНЕНИЕ Лимиты
   ПО ...;";
```

### Суть пояснения ИТС

- Оптимизатор СУБД плохо оценивает кардинальность подзапроса в соединении; после переписывания стороны соединений — физические таблицы (в т.ч. ВТ) → стабильнее план; накладные расходы на ВТ в типичном случае **малы**.

### Оговорка по замерам (Infostart 548778)

- В докладе приведён пример: виртуальная таблица **остатков**, отбор номенклатуры документа — **вложенный запрос** vs **вложенный запрос + ВТ**; по замеру автора в **«Консоли запросов для управляемого приложения 8.3»** (MS SQL) вариант с ВТ оказался **медленнее**: затраты на **создание** ВТ перевесили. Автор отмечает, что такое на практике **часто**.
- Вывод: рекомендация ИТС по ВТ остаётся **методикой по умолчанию** для проблемных соединений, но **подтверждать** её на своих данных; контекст замеров — [Infostart 548778](https://infostart.ru/1c/articles/548778/), **§10** ниже.

## 4. Соединения с виртуальными таблицами

### Правило (ИТС)

- Если **`СОЕДИНЕНИЕ` с виртуальной таблицей** (например, `РегистрНакопления.Товары.Остатки()`) и производительность **неудовлетворительна** — вынести обращение к виртуальной таблице в отдельный запрос с **`ПОМЕСТИТЬ`** во временную таблицу (**та же идея**, что для подзапроса в соединении).

### Пояснение

- Виртуальная таблица при трансляции в SQL **часто** разворачивается в **подзапрос** по нескольким физическим таблицам.

### ВТ после выноса виртуальной таблицы (Infostart)

- Если обращение к виртуальной таблице вынесено во ВТ по методике ИТС, всё равно оценить **размер** результата и **стоимость** материализации: при **небольшом** числе строк **`ИНДЕКСИРОВАТЬ`** по ВТ может дать одни накладные расходы без ускорения ([Infostart 548778](https://infostart.ru/1c/articles/548778/), подраздел **§4**).

## 5. Несоответствие индексов и условий

### Правило (ИТС)

- Для условий в **`ГДЕ`**, **`ПО`**, параметрах виртуальной таблицы, **`ИМЕЮЩИЕ`** должен быть **подходящий** индекс.

### Три требования к индексу

1. Индекс содержит **все** поля условия.  
2. Эти поля — в **начале** индекса.  
3. Поля идут **подряд**, без полей, не участвующих в условии, «между» ними.

### Автоиндексы (кратко из ИТС)

- по ссылке / УИД — объектные сущности;
- по **регистратору** — движения, подчинённые регистратору;
- по **периоду и всем измерениям** (порядок в конфигураторе) — итоги регистра накопления;
- по **периоду, счету и измерениям** — итоги регистра бухгалтерии.

Детали индексов — в статье ИТС по ссылке из оригинала (`kb.1c.ru` в материале 5842; при необходимости найти актуальный URL в ИТС).

- Дополнительные индексы по реквизитам — **осознанно**; не «на всякий случай»; **не** дублировать индекс по **первому измерению** регистра — подходит основной индекс итогов.

### Без подходящего индекса

- Сканирование → рост времени и **рост блокировок** → ниже параллелизм.
- В публичных материалах встречается позиция «подходящий индекс — не зона ответственности прикладного разработчика»; для запросов к типовой конфигурации это приводит к отборам по полям **без** индекса и сканированию ([Infostart 548778](https://infostart.ru/1c/articles/548778/)).

### Критерий отбора и индекс на СУБД (Infostart 548778)

- Автор доклада описывает приём **«неявного индексирования»**: при выборке из **табличной части документа** по полю (пример в докладе — **номенклатура**), если реквизит ТЧ **не** проиндексирован, добавление **критерия отбора** может привести к появлению на СУБД **подходящего покрывающего / некластеризованного** индекса под задачу; детали методики — на **ИТС** (в докладе отсылка к открытым материалам).
- **Ограничение:** использовать критерий отбора как основной механизм **выборки в тексте запроса** для номенклатуры из документов **разных видов** автор сравнивает не в пользу критерия — на СУБД строится **более сложный** SQL, чем при **функционально близком** варианте с **`ОБЪЕДИНЕНИЕ`** запросов; на слайде удаление критерия сильно замедляет запрос (роль критерия — в том числе **поддержка индекса**, а не любой текст запроса). Источник тезиса — [Infostart 548778](https://infostart.ru/1c/articles/548778/), настоящий раздел **§5**.

### Примеры на `РегистрНакопления.ТоварыНаСкладах.Остатки` (из ИТС)

**Запрос 1 — нарушено требование 2** (нет отбора по первому полю индекса — «Склад»):

```bsl
Запрос.Текст = "ВЫБРАТЬ
| ТоварыНаСкладахОстатки.Склад,
| ТоварыНаСкладахОстатки.Номенклатура,
| ТоварыНаСкладахОстатки.Качество
|ИЗ
| РегистрНакопления.ТоварыНаСкладах.Остатки(, Номенклатура = &Номенклатура) КАК ТоварыНаСкладахОстатки";
```

Варианты из статьи: проиндексировать «Номенклатуру» **или** поставить «Номенклатуру» **первой** в измерениях (осторожно к другим запросам).

**Запрос 2 — нарушено требование 3** (между «Склад» и «Качество» в метаданных есть «Номенклатура», в условии её нет):

```bsl
Запрос.Текст = "ВЫБРАТЬ
| ТоварыНаСкладахОстатки.Склад,
| ТоварыНаСкладахОстатки.Номенклатура,
| ТоварыНаСкладахОстатки.Качество
|ИЗ
| РегистрНакопления.ТоварыНаСкладах.Остатки(
| ,
| Качество = &Качество
| И Склад = &Склад) КАК ТоварыНаСкладахОстатки";
```

Варианты из статьи: добавить условие по «Номенклатуре»; убрать условие по «Качеству»; перенести «Номенклатуру» в реквизиты; поменять местами «Номенклатуру» и «Качество».

**Запрос 3 — корректно** (порядок условий в тексте **не обязан** совпадать с порядком полей в индексе):

```bsl
Запрос.Текст = "ВЫБРАТЬ
| ТоварыНаСкладахОстатки.Склад,
| ТоварыНаСкладахОстатки.Номенклатура,
| ТоварыНаСкладахОстатки.Качество,
| ТоварыНаСкладахОстатки.КоличествоОстаток
|ИЗ
| РегистрНакопления.ТоварыНаСкладах.Остатки(
| ,
| Номенклатура = &Номенклатура
| И Склад = &Склад) КАК ТоварыНаСкладахОстатки";
```

## 6. Логическое ИЛИ в условиях

### 6.1. Секция `ГДЕ`

- **`ИЛИ` в `ГДЕ`** может помешать индексам → сканирование, медленнее, больше блокировок.
- Замена — **`ОБЪЕДИНИТЬ ВСЕ`** отдельных запросов.

**Пример из ИТС:**

```1c
ВЫБРАТЬ Товар.Наименование ИЗ Справочник.Товары КАК Товар ГДЕ Артикул = "001" ИЛИ Артикул = "002"
```

**Замена:**

```1c
ВЫБРАТЬ Товар.Наименование ИЗ Справочник.Товары КАК Товар ГДЕ Артикул = "001"
|ОБЪЕДИНИТЬ ВСЕ
|ВЫБРАТЬ Товар.Наименование ИЗ Справочник.Товары КАК Товар ГДЕ Артикул = "002"
```

### 6.2. RLS и несколько ролей

- Одному пользователю **не** назначать **несколько** ролей с **RLS** — к запросу добавятся условия ролей через **`ИЛИ`** (та же проблема, даже если в тексте запроса **`ИЛИ`** нет).
- Решение: одна **комбинированная** роль с RLS **без** ИЛИ в условии.

### 6.3. `ИЛИ` в `ПО`

- Риск неоптимального плана; **универсальной** замены нет — пересмотр алгоритма.

## 7. Подзапрос в условии соединения (`ПО`)

### Правило (ИТС)

- **Не использовать** подзапрос в **`ПО`** — сильное замедление; на отдельных СУБД возможна **полная неработоспособность**.

### Пример из ИТС

```bsl
Запрос.Текст = "ВЫБРАТЬ
   | ОстаткиТоваров.Номенклатура КАК Номенклатура,
   | Цены.Цена КАК ЦенаПрошлогоМесяца
   |ИЗ
   | РегистрНакопления.ТоварыНаСкладах.Остатки(...) КАК ОстаткиТоваров
   | ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.Цена КАК Цены
   | ПО Цены.Номенклатура = ОстаткиТоваров.Номенклатура И
   | Цены.Период В (
   | ВЫБРАТЬ МАКСИМУМ(ЦеныПрошлогоМесяца.Период)
   | ИЗ РегистрСведений.Цена КАК ЦеныПрошлогоМесяца
   | ГДЕ ЦеныПрошлогоМесяца.Период < НАЧАЛОПЕРИОДА(ОстаткиТоваров.Период, МЕСЯЦ)
   | И ЦеныПрошлогоМесяца.Номенклатура = ОстаткиТоваров.Номенклатура
   | )
   | ГДЕ ОстаткиТоваров.Склад = &Склад";
```

### Пакет с ВТ (текст как в HTML ИТС)

```bsl
Запрос.Текст = "
  // Максимальные даты установки цен в прошлом периоде для данных номенклатур
   |ВЫБРАТЬ
   | ОстаткиТоваров.Номенклатура КАК Номенклатура,
   | МАКСИМУМ(Цены.Период) КАК Период
   |ПОМЕСТИТЬ ДатыПоНоменклатурам
   |ИЗ
   | РегистрНакопления.ТоварыНаСкладах.Остатки(...) КАК ОстаткиТоваров
   | ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.Цена КАК Цены
   | ПО Цены.Номенклатура = ОстаткиТоваров.Номенклатура И
   | Цены.Период < НАЧАЛОПЕРИОДА(ОстаткиТоваров.Период, МЕСЯЦ)
   | СГРУППИРОВАТЬ ПО ОстаткиТоваров.Номенклатура
   | ГДЕ ОстаткиТоваров.Склад = &Склад;
  
  // Выбрать данные по цене за найденный период
   |ВЫБРАТЬ
   | ДатыПоНоменклатурам.Номенклатура КАК Номенклатура,
   | Цены.Цена КАК ЦенаПрошлогоМесяца
   |ИЗ ДатыПоНоменклатурам
   | ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.Цена КАК Цены
   | ПО Цены.Номенклатура = ОстаткиТоваров.Номенклатура И
   | Цены.Период = ДатыПоНоменклатурам.Период
";
```

Во второй части пакета в HTML ИТС в условии **`ПО`** указано **`ОстаткиТоваров.Номенклатура`**, хотя во **`ИЗ`** только **`ДатыПоНоменклатурам`**; для исполняемого текста логично **`ДатыПоНоменклатурам.Номенклатура`**.

## 8. Обращение через точку от поля составного типа

### Правило (ИТС)

- **`ПолеСоставногоТипа.Реквизит`** порождает соединения **со всеми** таблицами типов состава → тяжёлый SQL.
- Не рекомендуется **`Регистратор.Дата`** и т.п. через точку — в любом месте запроса.

### Приёмы из статьи

- Сузить типы в метаданных; дублировать значение в реквизите; **`ВЫРАЗИТЬ`**; динамический текст запроса с нужным типом в **`ВЫРАЗИТЬ`**.

### Пример «плохо» (ИТС)

```bsl
Запрос.Текст = "ВЫБРАТЬ
| Продажи.Регистратор.Номер,
| Продажи.Регистратор.Дата,
| Продажи.Контрагент,
| Продажи.Количество,
| Продажи.Стоимость
|ИЗ
| РегистрНакопления.Продажи КАК Продажи
|ГДЕ ...";
```

В статье: **56** видов документов у регистратора → **56** левых соединений в SQL.

### Не усложнять без необходимости (Infostart 548778)

- В докладе отдельно разобран слайд: обращение к вложенному свойству составного **регистратора** (в примере — «Контроль цен»), при этом у вида документа **«Реализация товаров и услуг»** такого свойства **нет** — платформа **не** генерирует лишних соединений к этому виду документа. Автор предупреждает от **избыточной** «оптимизации» и переписывания, если лишние источники при данном тексте **и так не подключаются** ([Infostart 548778](https://infostart.ru/1c/articles/548778/), настоящий раздел **§8**).

### Уточнение из комментария к публикации Infostart (bulpi)

- Для поля **`Регистратор`** составного типа и вложенного реквизита вроде **`Регистратор.Контрагент`** в обсуждении к статье указано: **сначала типизировать** сам **`Регистратор`**, иначе риск лишних соединений на СУБД. Это согласуется с общим правилом ИТС (§8), но формулировка относится к **комментарию**, а не к основному тексту доклада.

## 9. Виртуальные таблицы: параметры vs `ГДЕ`

### Правило (ИТС)

- Условия виртуальной таблицы передавать в **параметры** таблицы, а не фильтровать только в **`ГДЕ`** вне параметров (функционально верно, план часто хуже).

**Пример «хуже» (ИТС):**

```bsl
Запрос.Текст = "ВЫБРАТЬ
| Номенклатура
|ИЗ
| РегистрНакопления.ТоварыНаСкладах.Остатки()
|ГДЕ
| Склад = &Склад";
```

**Пример «лучше» (ИТС):**

```bsl
Запрос.Текст = "ВЫБРАТЬ
| Номенклатура
|ИЗ
| РегистрНакопления.ТоварыНаСкладах.Остатки(, Склад = &Склад)";
```

### Период виртуальной таблицы остатков и «актуальные» данные (Infostart 548778)

- Для **актуальных** остатков отбор по **периоду** в параметрах виртуальной таблицы **может быть не нужен**.
- По смыслу доклада: остатки на **актуальную** отметку **без** использования параметра периода (в формулировке автора) дают чтение из **одного** источника — **таблица итогов**; если в параметрах указать **любой** «подходящий» период, в том числе **будущее** время **текущего дня**, при той же бизнес-семантике «актуальности», платформа может обращаться к **двум** источникам — **итоги и движения**, с **заметным** снижением производительности; автор называет это **очень частой** ошибкой.
- Конкретный синтаксис параметров виртуальной таблицы в докладе не приведён — сверять с **синтакс-помощником** и **замером** для вашей СУБД; см. раздел **§9** и **§10**.

## 10. Практика и замеры (Infostart 548778)

1. **Метрики на СУБД:** при оптимизации ориентироваться не только на «ощущения», а на **стоимость** выполнения запроса, **затраты ввода-вывода** и **процессор** (формулировка доклада).
2. **Инструмент:** обработка **«Консоль запросов для управляемого приложения 8.3»** с диска ИТС при сервере **MS SQL** — визуализация плана и затрат (как в примерах доклада).
3. **Среда:** подбирать решение в **тех же условиях**, в которых работает система (объём данных, нагрузка, настройки СУБД).
4. В тексте публикации [Infostart 548778](https://infostart.ru/1c/articles/548778/) **не** воспроизведён перечень **восьми пунктов** со слайда доклада — при необходимости смотреть исходные материалы публикации или видео/скриншоты доклада.

## 11. Чеклист самопроверки перед выдачей кода

1. Нет ли **`СОЕДИНЕНИЕ` с подзапросом** или медленного соединения с **виртуальной таблицей** без ВТ?
2. Для каждого **`ГДЕ` / `ПО` / параметров ВТ / `ИМЕЮЩИЕ`** проверены три условия соответствия **индексу**?
3. Нет ли **`ИЛИ` в `ГДЕ`** там, где нужны индексы? Учтены ли **RLS** и несколько ролей?
4. Нет ли **подзапроса в `ПО`**?
5. Нет ли лишних **точек от составного типа** без **`ВЫРАЗИТЬ`** / дублирования / сужения типов? Нет ли переписывания «на всякий случай», если при данном тексте лишние типы **не** подключаются (Infostart)?
6. Условия **виртуальных таблиц** — в **скобках параметров**, а не только в **`ГДЕ`**? Для **остатков на актуальную** дату проверен смысл параметров **периода** (итоги vs итоги+движения) и замер (Infostart)?
7. После переноса в **ВТ** / **`ИНДЕКСИРОВАТЬ`** — сравнение **с исходным** вариантом на реальных объёмах; нет ли регресса из‑за накладных на малой выборке (Infostart)?
8. Для выборок из **ТЧ документа** по неиндексированному полю: рассмотрен ли **критерий отбора** как способ индекса на СУБД (ИТС + Infostart)? Для **разных видов** документов не выбран ли критерий в запросе там, где **`ОБЪЕДИНЕНИЕ`** даёт проще SQL (Infostart)?
9. После правок запроса при **ожиданиях** других сеансов — чеклист **`docs/excessive_locks_audit_1c.md`**.
10. Есть ли **замер** (план, I/O, CPU, время) в рабочей или близкой среде (**§10**, ИТС «Консоль запросов УП 8.3» для MS SQL)?
