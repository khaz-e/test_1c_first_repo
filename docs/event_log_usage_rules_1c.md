# Журнал регистрации 1С: правила записи событий из кода

## 1. Назначение документа

Документ задаёт правила **записи в журнал регистрации** из встроенного языка (`ЗаписьЖурналаРегистрации`) при разработке и ревью кода расширения. Цель — предсказуемая диагностика для администратора без «шума», дублирования событий в одной записи и типичных проблем с производительностью при анализе журнала.

**Источник методики:** [ИТС, Система стандартов: Использование Журнала регистрации](https://its.1c.ru/db/v8std/content/498/hdoc) (**#std498**).

**Связанные документы проекта:**

- Явные транзакции и запись при ошибках в транзакционном контексте: **`docs/transactions_audit_1c.md`** (ИТС [783](https://its.1c.ru/db/v8std/content/783/hdoc)).

**Дополнительно по стандарту 498 (локализация текстов в коде):**

- [ИТС #std761 — интерфейсные тексты в коде](https://its.1c.ru/db/v8std/content/761/hdoc) (в тексте стандарта 498 указана связь с локализацией типов событий).

## 2. Правила для ИИ-агента: порядок проверки

1. **Зачем пишем.** Запись из кода — когда администратору нужна **дополнительная диагностика**, которую платформа **не пишет автоматически** (интерактив и фоновые/регламентные задания).
2. **Одна запись — одно событие.** Не склеивать цепочку шагов в один комментарий; столько вызовов `ЗаписьЖурналаРегистрации`, сколько отдельных событий.
3. **Тип события.** Строковый идентификатор, группировка **`Группа.Событие`**; в строку типа **не** включать данные из ИБ/внешних систем — их выносить в параметры **«Метаданные»** и **«Данные»** метода `ЗаписьЖурналаРегистрации`.
4. **Локализация.** Текст типа события — **локализуемый**, на **основном языке конфигурации** (как в примерах стандарта с `НСтр` и `КодОсновногоЯзыка`).
5. **Уровень.** Соответствие смыслу: `Ошибка` / `Предупреждение` / `Информация` / `Примечание` (см. §4).
6. **Комментарий.** Неструктурированное пояснение к **одному** событию; без полей неограниченного размера; ориентир объёма — **до 10 Кб**.
7. **Исключения.** Для подробностей об ошибке в комментарии использовать **`ОбработкаОшибок.ПодробноеПредставлениеОшибки(ИнформацияОбОшибке())`**.
8. **Производительность.** Не строить **быстрые** прикладные сценарии на **выборках из журнала**; для нагруженного протоколирования — отдельный регистр или платформенные средства (в стандарте пример — **`МенеджерФоновыхЗаданий`**). То же при отчётах по журналу.

---

## 3. Назначение журнала (смысл)

Журнал регистрации хранит события работы пользователей с информационной базой. При администрировании журнал анализируют в разных разрезах: какие события были в момент времени, что делал конкретный пользователь и т.д.

---

## 4. Уровни важности события

| Уровень (`УровеньЖурналаРегистрации`) | Когда использовать |
|--------------------------------------|---------------------|
| `Ошибка` | критичные события и сбои |
| `Предупреждение` | потенциальные проблемы и нефатальные ошибки |
| `Информация` | успешные действия |
| `Примечание` | менее значимые сообщения |

---

## 5. Тип события: группировка и контент

Используйте **строковый идентификатор** типа события. Группируйте по функциональному признаку: **`НазваниеГруппыСобытий.НазваниеСобытия`**.

**Предпочтительно (примеры из стандарта):**

- `Поручения.Уведомление о новых задачах`
- `Поручения.Уведомление о зависших задачах`

**Хуже — без группы (примеры из стандарта):**

- `Уведомление о новых задачах`
- `Уведомление о зависших задачах`

**В тип события не включайте** конкретные данные из ИБ или внешних систем.

**Некорректно как тип события (примеры из стандарта):**

- `Не удалось отменить проведение документа <Реализация КП00-00002 от 11>`
- `В информационной базе не найден сотрудник Найденова, банковская карта 3333`

Метаданные события передавайте в параметр **«Метаданные»**, данные — в параметр **«Данные»** метода `ЗаписьЖурналаРегистрации`.

Типы событий делайте **более общими**, например (из стандарта):

- `Проведение документов.Не удалось отменить проведение`
- `Сотрудники.В информационной базе не найден сотрудник`

---

## 6. Комментарий: один комментарий — одно событие, ограничение размера

Комментарий — **неструктурированный** текст о событии. Не объединяйте в одном комментарии несколько событий. Не помещайте в комментарий текст неизвестного объёма (содержимое файлов, ответы веб-сервисов и т.п.); ориентир по размеру — **до 10 Кб**.

**Неправильно (пример из стандарта):** один комментарий содержит целую последовательность разных событий:

```text
[01.01.2010 00:00:01] Начало инициализации обмена данными...
[01.01.2010 00:00:02] Окончание инициализации обмена данными...
[01.01.2010 00:00:03] Начало процесса обмена данными...
[01.01.2010 00:00:04] Начало записи изменений...
[01.01.2010 00:00:05] Окончание записи изменений...
[01.01.2010 00:00:06] Окончание процесса обмена данными...
[01.01.2010 00:00:07] Выполнено, выгрузка данных...
```

Пишите столько записей журнала, сколько событий реально произошло.

---

## 7. Исключения в комментарии

Для записи данных об исключении используйте (как в стандарте):

```bsl
ОбработкаОшибок.ПодробноеПредставлениеОшибки(ИнформацияОбОшибке())
```

## 8. Пример: старт, успех, ошибка (из стандарта)

Регистрация дополнительных событий подсистемы — отдельные записи на «начато», «завершено» и запись в `Исключение` с уровнем `Ошибка` и подробным представлением ошибки:

```bsl
Попытка
    ЗаписьЖурналаРегистрации(НСтр("ru = 'Мой механизм.Действие с возможной ошибкой'", КодОсновногоЯзыка),
        УровеньЖурналаРегистрации.Информация, , ,
        НСтр("ru = 'Начато действие'"));

    ДействиеСВозможнойОшибкой(ОбъектДействия);

    ЗаписьЖурналаРегистрации(НСтр("ru = 'Мой механизм.Действие с возможной ошибкой'", КодОсновногоЯзыка),
        УровеньЖурналаРегистрации.Информация, , ,
        НСтр("ru = 'Завершено действие'"));

Исключение
    ЗаписьЖурналаРегистрации(НСтр("ru = 'Мой механизм.Действие с возможной ошибкой'", КодОсновногоЯзыка),
        УровеньЖурналаРегистрации.Ошибка, , ,
        НСтр("ru = 'Во время выполнения действия произошла неизвестная ошибка.'") + Символы.ПС
        + ОбработкаОшибок.ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
КонецПопытки;
```

Переменная `КодОсновногоЯзыка` должна содержать код языка хранения данных (в стандарте 498 отсылка к **#std784**, п. 1).

---

## 9. Производительность и выборки из журнала

Не используйте выборки из журнала регистрации там, где критична **высокая скорость**: при росте объёма журнала скорость выборки снижается пропорционально объёму.

Для протоколирования производительных сценариев предпочтительнее:

- отдельный регистр;
- специализированные объекты платформы, например **`МенеджерФоновыхЗаданий`** (история фоновых заданий).

То же учитывать при разработке **отчётов** по журналу регистрации.
